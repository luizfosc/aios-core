## Story

**As a** user,
**I want** the system to automatically validate the quality of the edited document comparing it against the raw input,
**so that** I can be confident nothing substantive was lost and the speaker's DNA was preserved.

## Acceptance Criteria

1. Raw vs edited comparison: content preservation percentage calculated (target >= 98%)
2. DNA preservation score (1-10) based on catchphrase retention, vocabulary consistency, tone fidelity
3. Structural coherence score (1-10) based on section flow, heading hierarchy, zone consistency
4. List of all removed excerpts with reason for each removal (auditable trail)
5. List of all excerpts marked `[REVISAR]` with location and context
6. Quality report generated in YAML format per architecture spec (`quality-report.yaml`)
7. Verdict: PASS (overall >= 8.0), CONCERNS (6.0-7.9), FAIL (< 6.0)

## Dev Notes

### Quality Report Schema (from architecture doc)

```yaml
quality_report:
  source: "raw/palestra-dia1.md"
  output: "output/palestra-dia1-masterpiece.md"
  processed_at: "2026-02-22T15:45:00Z"
  scores:
    content_preservation: 98.2
    dna_preservation: 9.1
    structural_coherence: 8.8
    overall: 8.7
  content_analysis:
    words_original: 42350
    words_final: 41580
    words_removed: 770
    words_removed_noise: 680
    words_removed_duplicates: 90
    words_marked_review: 12
  removed_excerpts:
    - text: "[murmurios inaudiveis]..."
      reason: "noise"
  review_excerpts:
    - text: "Alguem pode fechar a porta?"
      location: "Section 3, paragraph 12"
      reason: "Possibly relevant stage direction"
  dna_analysis:
    catchphrases_found: ["galera", "olha so", "percebam", "sacou?"]
    catchphrases_preserved: 4
    catchphrases_lost: 0
    tone_consistency: "high"
    formality_level_maintained: true
  verdict: "PASS"
```

### Scoring Weights

| Score | Weight | Description |
|-------|--------|-------------|
| content_preservation | 40% | Raw vs edited word count + semantic check |
| dna_preservation | 35% | Catchphrases, vocabulary, tone, formality |
| structural_coherence | 25% | Headings, flow, zone consistency |

### Verdict Thresholds

| Overall Score | Verdict | Action |
|---------------|---------|--------|
| >= 8.0 | PASS | Proceed to merge/delivery |
| 6.0 - 7.9 | CONCERNS | Flag for human review, proceed |
| < 6.0 | FAIL | Return to transcript-editor for re-edit |

### Speaker Detection Threshold (PO recommendation)

For FR6 (speaker detection), accuracy is best-effort. Low-confidence detections should be marked `[REVISAR]` rather than silently dropped or incorrectly attributed.

- Verify verdict thresholds work (inject low-quality edit → expect FAIL)

### Decision Log

**Decision 1: Input structure**
- Changed from per-chunk validation to full-file validation
- Rationale: Quality validation needs global view (all chunks aggregated) to produce final quality-report.yaml
- Input: raw file + all edited chunks + analysis-map → Output: single quality-report.yaml

**Decision 2: Analysis-map integration**
- Added analysis-map as required input for catchphrase extraction
- Rationale: Cannot validate DNA preservation without knowing expected catchphrases from speakers
- Source: analysis-map.yaml → speakers[].characteristics

**Decision 3: Checklist granularity**
- Conversion checklist: 6 items (focused on ingestion quality)
- Editorial checklist: 10 items (comprehensive DNA + structure validation)
- Rationale: Different stages have different quality concerns, separate checklists clearer

**Decision 4: Critical items marking**
- Editorial checklist marks 4 items as ⚠️ CRITICAL (non-negotiable)
- Rationale: Some rules have zero tolerance (paraphrasing, catchphrase loss) — must be visually distinct
