### Example 3: Analytics Integration Design

```markdown
# ML + Rules Integration: Fraud Detection Decision

## Decision Context
**Decision:** Transaction Fraud Assessment
**Owner:** Fraud Prevention Director
**Frequency:** 2M transactions/day
**Current State:** Rule-based only, 15% false positive rate

## Integration Architecture

### Predictive Model Component
- **Model:** XGBoost classifier (fraud probability score)
- **Inputs:** 45 transaction features (amount, location, device, velocity, merchant category, etc.)
- **Output:** Fraud probability (0.0 - 1.0)
- **Refresh:** Real-time inference via REST API

### Decision Table Component (DMN)
**Decision:** Block Transaction
**Inputs:**
- ML fraud probability score
- Transaction amount
- Merchant risk tier (from business rule lookup)
- Customer VIP status (from CRM)

**Decision Logic:**

| Fraud Score | Amount | Merchant Tier | VIP Status | Action |
|-------------|--------|---------------|------------|--------|
| >= 0.8 | - | - | - | BLOCK |
| 0.5 - 0.79 | > $1000 | High Risk | false | BLOCK |
| 0.5 - 0.79 | > $1000 | High Risk | true | CHALLENGE |
| 0.5 - 0.79 | <= $1000 | - | - | CHALLENGE |
| 0.3 - 0.49 | > $5000 | - | - | CHALLENGE |
| < 0.3 | - | - | - | ALLOW |

## Integration Flow
```
Transaction Event
  ↓
ML Model API (fraud score)
  ↓
DMN Decision Service (combines score + business rules)
  ↓
Action (ALLOW | BLOCK | CHALLENGE)
```

## Business Rule Governance
- **ML Model:** Data Science team owns retraining (weekly)
- **Decision Table:** Fraud Prevention team owns thresholds (monthly review)
- **Merchant Risk Tier:** Risk Ops team maintains lookup table

## Expected Impact
- Reduce false positives from 15% to 8% (via ML precision)
- Maintain business control over VIP handling (via decision table)
- Enable A/B testing on threshold changes without model redeployment
```
