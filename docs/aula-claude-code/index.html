Situações que todo dev da Ensinio já viveu pelo menos uma vez por semana.

O problema não é vocês. O Claude Code, sem configuração, é um assistente genérico. Ele não sabe nada sobre o Ensinio Club, Pay, Stream, AI, sobre as regras de negócio, sobre o padrão de código do time. A cada sessão, ele começa do zero.

Capítulo 02

Onde você está hoje?

Você já percebeu que passa mais tempo GERENCIANDO o Claude do que codando de verdade? "Explica isso", "não, esquece", "volta", "corrige ali". Você virou babá de IA. E se ele soubesse trabalhar sozinho?

1Básico

Você dita, a IA digita

Tipo Copilot turbinado — economiza dedos, mas não economiza cabeça.

Cria um componente React com um formulário de matrícula pro Ensinio Club

  • Dita a implementação exata ("usa useState", "faz um map")
  • Revisa manualmente cada linha
  • Nenhum conhecimento é compartilhado entre devs
2Inter

Você descreve o resultado, a IA escolhe o caminho

O Claude já sabe o projeto. Você foca no O QUE, não no COMO.

O checkout do Pay precisa validar CPF antes de processar o Pix. Segue o padrão de validação que já existe no projeto.
  • CLAUDE.md configurado — a IA já sabe a stack, convenções e regras
  • Slash commands: /review, /test, /fix
  • Dá contexto: "segue o padrão", "como o arquivo X"
  • Configuração commitada no repo — todo dev herda
3Pro

Você define o problema, agentes resolvem

Você vira o gestor. Agentes especializados fazem o trabalho operacional.

Precisamos que a Anamnese do Club direcione o aluno automaticamente para a trilha certa. Quando ele responde o questionário de perfil, os pesos geram um score e matriculam na trilha ideal. Se empatar, mostra as opções.
  • Define O QUE precisa ser feito, não COMO
  • Agentes criam a story, arquitetam, implementam, testam
  • Verificação automática (lint, test, typecheck) antes de qualquer push
  • Histórico rastreável via git log + story markdown

Hoje vamos do Nível 1 pro Nível 2.
Nível 3 é consequência natural.

Não tente pular. Um CLAUDE.md bem feito já muda completamente a experiência. Depois, commands e rules. Quando isso virar rotina, você está pronto pro próximo passo.

Capítulo 03

O segredo: dar contexto uma vez só

Você já perdeu 20 minutos explicando "usa Next.js 14", "com Prisma", "não, sem tRPC"... pra descobrir que ele esqueceu tudo na próxima sessão? Dá pra ensinar uma vez só.

Pensa assim: O CLAUDE.md é o crachá de acesso do projeto. Sem ele, o Claude é um estagiário novo que chega todo dia e pergunta "cadê o banheiro?". Com ele, é o dev que já passou pelo onboarding: sabe onde tudo fica, conhece as regras da casa.

Slash commands compartilhados review.md .... /review — checklist de code review fix.md ....... /fix — roda lint+test e corrige rules/ ......... Regras por contexto (injetadas automático) payment.md ... Regras quando mexe em código de pagamento api.md ....... Convenções de API REST
# CLAUDE.md — carregado em TODA sessão # Imagine: o que você explicaria a um dev novo? ## Stack - Next.js 14, App Router, TypeScript strict - Tailwind CSS + Radix UI - Prisma ORM + PostgreSQL - Stripe (Pay), Mux (Stream), Cloudflare R2 ## Produtos - Club: área de membros (cursos, quizzes, tarefas) - Pay: checkout integrado (Pix, cartão, boleto) - Stream: vídeo hosting próprio - AI: agentes SDR e customizados (vendas, atendimento) - Engage: comunidades segmentadas + gamificação ## Comandos npm run dev # Porta 3000 npm test # Jest + Testing Library npm run lint # ESLint + Prettier ## Regras - NUNCA usar "any" — tipar tudo - NUNCA commitar .env ou credenciais - Toda API route valida input com Zod - Testes junto ao arquivo: Component.test.tsx
# .claude/commands/review.md # Qualquer dev do time digita /review e usa Faça um code review do diff atual: 1. Segurança: injection, XSS, dados expostos 2. Performance: N+1 queries, renders desnecessários 3. Tipos: any, assertions, casts inseguros 4. Regras de negócio: validações do Pay/Club CRITICAL — bloqueia merge WARNING — deveria corrigir INFO — sugestão # Um dev configura. Todo o time usa. Sempre.
# .claude/rules/payment.md # Ativado AUTOMATICAMENTE quando Claude edita # qualquer arquivo em src/services/payment/ Ao trabalhar com código de pagamento: 1. SEMPRE validar CPF/CNPJ antes de processar 2. Pix: timeout de 15 min, depois cancela 3. Cartão: retentativa em 25 dias 4. Boleto: vencimento D+3 úteis 5. NUNCA logar dados de cartão (PCI compliance) # Você não precisa lembrar essas regras. # O Claude segue automaticamente.

Slash commands = atalhos de teclado pra prompts. Sabe Cmd+S pra salvar? Imagine /fix que roda lint+test+corrige tudo. Rules = avisos de segurança automáticos. Se você mexe em payment.ts, o Claude já sabe "NUNCA logar dados de cartão" sem você lembrar.

Capítulo 04

A diferença na prática

O mesmo pedido. Dois resultados completamente diferentes.

Bug no Ensinio Pay

Sem config
> o checkout ta bugado Claude: Pode me contar mais? Qual framework? Como é o checkout? Qual gateway de pagamento? 5 mensagens depois... ainda alinhando contexto.
Com CLAUDE.md
> o checkout do Pay dá 500 quando paga com Pix Claude já sabe: Next.js, Prisma, estrutura do Pay, Pix com timeout de 15min, validação de CPF... Lê o arquivo, acha o bug, corrige em 30 segundos.

Feature: Follow-up automático no Ensinio AI

Prompt vago
> cria um sistema de follow-up Claude: Follow-up de quê? Emails? CRM? Vendas? Qual intervalo? Qual canal de comunicação? Resultado genérico, sem relação com o Ensinio AI.
Prompt com contexto
> O agente SDR do Ensinio AI precisa mandar follow-up se o lead não responder em 24h. Max 3 tentativas, tom consultivo. Sem insistência agressiva. Resultado preciso, integrado com o agente SDR existente.

O padrão de ouro pra pedir qualquer coisa

CONTEXTO (o que tá acontecendo) + ESPERADO (o que deveria acontecer) + LIMITES (o que NÃO precisa agora)

Templates prontos — copie e adapte

Bug

O QUE + QUANDO + ESPERADO + PISTA

"POST /api/pay/pix retorna 500 quando CPF tem pontuação. Esperado: aceitar com ou sem máscara. Erro está na validação."

Feature

O QUE + CENÁRIOS + O QUE NÃO PRECISA

"Notificar alunos quando: nova aula, resposta no feed, certificado disponível. In-app por agora, push depois."

Refactor

ESTADO ATUAL + ESTADO DESEJADO + RESTRIÇÕES

"payment.ts tem 800 linhas misturando Pix+cartão+boleto. Separar em 3 + orquestrador. Testes devem continuar passando."

Capítulo 05

E quando você quiser ir além?

Até aqui, você configurou o CLAUDE.md, criou commands, tá economizando tempo de verdade. Mas repara: ainda é VOCÊ que lembra de rodar lint. VOCÊ que cria o PR. VOCÊ que decide a ordem das coisas. E se cada etapa tivesse um responsável — e você só validasse o resultado final?

Analogia: Imagina que você é o diretor de um restaurante. Hoje, você é o chef, o garçom, o caixa e o faxineiro. O AIOS é como contratar uma equipe: cada pessoa faz só o que manja. Você continua decidindo o cardápio — mas não precisa mais fritar o ovo.

Antes e Depois — o que muda

Você faz tudo sozinhoCada etapa tem um responsável
Você lembra de rodar lint antes do commitO inspetor de qualidade roda automático (lint + test + typecheck)
Você cria a descrição do PR manualmenteO responsável por deploy cria PR com descrição baseada na tarefa
Você decide cada passo da implementaçãoFluxo definido: documentar → implementar → validar
Qualquer prompt pode dar push por acidenteSó o responsável por deploy pode push — catraca de segurança

Quem é quem no time?

Você é o líder técnico. Cada agente é um especialista que só faz o que manja. Ninguém invade a área do outro.

@sm — O Documentador

Escreve a "receita" (story) com tudo que precisa ser feito: critérios, cenários, limites. Como um analista de requisitos que não esquece nada.

@po — O Validador

Confere se a receita está completa. "Cobriu o cenário do Pix? E se o CPF vier sem pontuação?" Só libera quando tá redondo.

@dev — O Implementador

Pega a receita validada e codifica. Faz commit local, atualiza a story. É o único que escreve código de produção.

@qa — O Inspetor

Roda lint, typecheck e testes. Se algo falhar, devolve pro dev com feedback específico. Nada passa sem aprovação dele.

@devops — O Porteiro

O ÚNICO que pode dar push e criar Pull Request. Nenhum outro agente tem essa permissão. É a catraca de segurança do projeto.

Exemplo real: Parcelamento Inteligente no Pay

Recuperador de Vendas — Pay
Você: Quando o cartão do aluno é recusado por limite, o Pay precisa oferecer parcelamento recorrente automático. Ex: R$1.200 em 12x vira 12 cobranças de R$100, consumindo só R$100 do limite por vez. Se ainda não tiver limite, volta pro checkout. # Cada especialista faz a sua parte: @sm Documenta: cenários, regras de retry, limites de parcela @po Valida: "e se o cartão é recusado na 3a parcela?" @dev Implementa o fluxo no checkout + lógica de reprocessamento @qa Verifica: lint ok, testes passam, PCI compliance ok @devops Push + Pull Request pro time revisar Você definiu O QUE. Cada especialista resolveu O COMO. Importante: você ainda valida o resultado final. O AIOS acelera, não substitui seu julgamento.

Capítulo 06

Quick Wins — aplique amanhã

4 ações concretas que qualquer dev da Ensinio pode fazer agora.

5m

Crie o CLAUDE.md do seu projeto

Rode /init no Claude Code — ele gera automaticamente.

  • Stack (Next.js, Prisma, etc)
  • Comandos (dev, test, lint, build)
  • Convenções (naming, pastas, imports)
  • Regras do que NUNCA fazer
15m

Crie 3 slash commands

Automatize o que você repete toda semana.

  • /review — Code review com checklist padrão
  • /test — "Cria testes pro arquivo atual"
  • /fix — "Roda lint e typecheck, corrige os erros"
30m

Crie rules pro que se repete

Regras que ativam automaticamente quando necessário.

  • .claude/rules/payment.md — regras do Pay (PCI, CPF, etc)
  • .claude/rules/api.md — convenções de API REST
  • .claude/rules/club.md — regras do Club (matrícula, progresso)
1h

Mude como você pede

A maior alavanca de todas.

  • Pare de ditar implementação. Descreva o resultado.
  • Sempre: O QUE acontece HOJE vs O QUE DEVERIA acontecer
  • Diga o que NÃO precisa (limita o escopo)
  • Peça pra seguir os padrões existentes do projeto

Checklist de Setup

  • Rodar /init pra gerar CLAUDE.md automaticamente
  • Revisar e ajustar o CLAUDE.md gerado (stack, regras, comandos)
  • Criar pelo menos /review em .claude/commands/
  • Commitar o .claude/ no repo (time inteiro herda)
  • No próximo bug: descrever com CONTEXTO + ESPERADO + LIMITES
  • Parar de microgerenciar — delegar e validar resultado
Capítulo 07

Como isso gruda na cabeça?

Saber não é fazer. A diferença entre "entendi" e "uso todo dia" está em como você pratica. Não precisa de força de vontade — precisa de estratégia.

1

Comece ridiculamente pequeno

Hábito funciona como exercício: o que importa não é a intensidade, é a consistência. Não tente "configurar tudo hoje". Comece com UMA coisa: rode /init. Amanhã, crie UM command. Depois de uma semana, já virou rotina.

2

Note quando funcionar

Seu cérebro registra quando o resultado é melhor do que esperava. A primeira vez que o Claude corrigir um bug em 30s (em vez de 20min) — pare e note isso. Essa surpresa positiva é o que cria o hábito de voltar.

3

Calibre o desafio

Difícil demais = frustração. Fácil demais = tédio. O ponto ideal: tarefas que você já sabe fazer, mas que o Claude faz mais rápido. Comece por bugs simples, depois features pequenas. Vá subindo aos poucos.

4

Compartilhe com o time

Quando um dev vê outro economizando 30min/dia com /review, ele quer também. Manda no canal: "olha esse comando que criei, corta metade do meu tempo de code review". Adoção por contágio é mais forte que treinamento.

A sequência que funciona: Semana 1: /init + CLAUDE.md. Semana 2: criar /review e /fix. Semana 3: rules para payment e API. Semana 4: mudar como você pede (CONTEXTO + ESPERADO + LIMITES). Em 30 dias, você não volta mais pro Nível 1.

Não é sobre aprender uma ferramenta.
É sobre mudar como você trabalha.

A ferramenta é o Claude Code. O aprendizado real é parar de microgerenciar, definir critérios claros e confiar no processo. Isso serve pra IA e pra vida.
