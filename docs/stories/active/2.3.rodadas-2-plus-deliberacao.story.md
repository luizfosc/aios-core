# Story 2.3: Rodadas 2+ — Deliberação com Exposição

## Description

As a **operador da skill politica-app**,
I want **que nas rodadas 2+, cada agente receba o sumário dos argumentos anteriores e possa revisar sua opinião**,
So that **a simulação capture a dinâmica de formação de opinião quando pessoas são expostas a argumentos de outros**.

## Acceptance Criteria

- [x] AC1: Template `templates/deliberation-round-n.md` implementado com injeção de sumário + distribuição percentual no prompt
- [x] AC2: Template instrui o agente a considerar argumentos dos outros COM BASE em quem ele é (perfil cognitivo), não em lógica abstrata
- [x] AC3: Output YAML de cada agente inclui campos adicionais: `opinion_changed` (true|false), `change_trigger` (string — qual argumento causou a mudança), `previous_position` (a_favor|contra|neutro|indeciso)
- [x] AC4: Mudança de opinião é governada pelo perfil do agente: `opinion_flexibility`, `social_proof`, `authority`, `processing_style` (conforme tabela da seção 5.2 da arquitetura)
- [x] AC5: Mesmo mecanismo de spawn paralelo em batches da Story 2.1 — reutilizar lógica de batching e tratamento de falhas
- [x] AC6: Outputs salvos em `output/{run}/rounds/round-{N}.yaml` com mesma estrutura do round-1, adicionando os campos de mudança
- [x] AC7: Rodada 3 (consolidação) recebe sumário ATUALIZADO + distribuição de sentimento, conforme seção 5.1 da arquitetura

## Dev Notes

- **Referências diretas:**
  - Arquitetura seção 5.1 (Rodada 2: Exposição e Revisão + Rodada 3: Consolidação)
  - Arquitetura seção 5.2 (Mecanismo de Mudança de Opinião — tabela de fatores)
  - Spec seção 3.8 (Modelo de Formação de Opinião)
  - Spec FR12, FR13, FR14
- **Template de rodada 2+** deve incluir:
  - Sumário dos argumentos (output do summarizer)
  - Distribuição percentual atual
  - Instrução explícita: "Considere esses argumentos. Com base em QUEM VOCÊ É..."
  - Perguntas direcionais: "Algum argumento te fez repensar? Qual e por quê? Sua posição mudou?"
- **Detecção de mudança:** O campo `change_trigger` é essencial para o cálculo de `impact_score` na agregação (Epic 3)
- **Rodadas 4-5:** Opcionais, mesmo padrão da rodada 3. Configurável via `config.yaml` `rounds`
- **Não re-spawnar sub-agente com contexto anterior:** Agentes são stateless. O "estado" é simulado via injeção do sumário e distribuição
