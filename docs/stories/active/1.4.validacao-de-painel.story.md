# Story 1.4: Validação de Painel

## Description

As a **operador da skill politica-app**,
I want **rodar um dry-run de geração de painel e validar que os perfis são diversos e bem formados**,
So that **eu tenha confiança de que a base do sistema funciona antes de avançar para o motor de deliberação**.

## Acceptance Criteria

- [x] AC1: Dry-run executado usando preset `brasil-geral` com 10 agentes (metade do painel completo, para economia de tokens)
- [x] AC2: `panel.yaml` gerado e salvo em `output/{timestamp}-dry-run-painel/panel.yaml` com 10 perfis completos
- [x] AC3: Nenhum par de agentes tem perfil idêntico — validação manual comparando campos `demographic.region`, `demographic.class`, `psychographic.values`, `deliberative.processing_style`
- [x] AC4: Todos os 10 perfis contêm todos os campos obrigatórios do schema: `id`, `name`, `demographic` (7 subcampos), `psychographic` (4 subcampos), `deliberative` (4 subcampos)
- [x] AC5: Pelo menos 3 regiões distintas representadas nos perfis gerados
- [x] AC6: Pelo menos 2 `processing_style` diferentes (emocional e analítico) presentes no painel
- [x] AC7: Instruções de geração de painel documentadas no SKILL.md (seção de Assembly) para que o dry-run seja reprodutível

## Dev Notes

- **Referência:** Spec Story 1.4 — "Dry-run de geração de painel com 10 agentes usando preset brasil-geral"
- **Validação é manual** (restrição CON-10: sem testes unitários tradicionais)
- **Como rodar:** O SKILL.md deve conter instruções claras de como executar apenas a fase de Assembly isoladamente
- **O que verificar:** Diversidade de perfis é o critério principal. Se 10 agentes gerarem perfis muito similares, o template de persona precisa de ajustes ANTES de prosseguir para o Spike
- **Spec FR9:** "Armazenar perfis gerados em panel.yaml para auditoria e reuso"
- **Output:** O panel.yaml gerado serve como input para o Spike (spike-1) que vem logo em seguida
